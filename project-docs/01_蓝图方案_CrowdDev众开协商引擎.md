# CrowdDev 众开协商引擎 — 蓝图方案文档

**版本：** v1.0  
**日期：** 2026年3月  
**定位：** 面向复杂系统开发的 AI 多角色协商设计工具

---

## 一、核心理念

### 1.1 问题定义

传统产品开发模式中，系统设计由产品经理主导，用户反馈在上线后才能获得。这导致三个结构性问题：

**设计盲区**：产品经理无法同时深入理解所有角色的工作场景和利益诉求。调度员关心操作效率，一线人员关心响应速度，系统管理员关心审计合规——这些诉求之间的冲突往往在开发完成后才暴露。

**反馈滞后**：从设计到开发到测试到上线，反馈周期以月计。等发现问题时修改成本已经很高。

**知识流失**：设计讨论中的关键决策、妥协点和遗留分歧通常散落在会议纪要和聊天记录中，无法被系统化沉淀和复用。

### 1.2 众开模式（CrowdDev）

CrowdDev 提出一种新的开发范式——**众开模式**：产品不是由某个人设计出来的，而是由 AI 模拟的多角色群体通过结构化协商共同塑造出来的。

核心主张：

- **角色即 Agent**：系统中的每个用户角色被建模为一个有专业背景、性格特征和利益诉求的 AI Agent，它从自己的立场出发参与设计讨论
- **冲突即价值**：角色之间的利益冲突不是需要避免的问题，而是产品设计中最有价值的信号——冲突暴露的是真实的设计张力
- **协商即迭代**：通过结构化的协商流程（表态→分析→辩论→共识），将冲突转化为具体的设计决策
- **演化即产品**：角色档案、协商记录和共识决策构成系统的"设计基因"，随着每一轮讨论持续演化

### 1.3 从设计工具到数字员工

CrowdDev 的演进路径是一个自然的连续体：

```
设计阶段的模拟角色 ──→ 测试阶段的验收角色 ──→ 运营阶段的数字员工
         │                      │                       │
    参与功能设计           验证实现质量            替代部分人工操作
    暴露设计冲突           发现体验偏差            基于数据优化系统
    积累角色知识           完善角色模型            成为真正的 Agent
```

这意味着在 CrowdDev 中参与设计讨论的"调度员 Agent"，积累了所有讨论、反馈和决策偏好后，可以直接演化为运营阶段的"数字调度员"——它对系统的理解深度远超一个后期接入的通用 AI 助手。

## 二、战略定位

### 2.1 目标用户

**首要用户：** 开发复杂人机协同系统的技术团队（如城市安防、工业运维、智慧医疗等领域）。这些系统的共同特征是：涉及多个专业角色、角色之间存在天然的利益博弈、人机协同的边界需要精心设计。

**次要用户：** 政府/企业客户的运营团队。当 CrowdDev 中的角色 Agent 进入运营阶段后，它成为客户组织中每个岗位的"数字员工伴侣"，帮助优化工作流程。

**远期用户：** 任何需要多角色协同设计的产品团队。CrowdDev 的核心逻辑（角色建模→冲突发现→协商决策）可以泛化到所有产品设计场景。

### 2.2 切入策略

以 CityMatrix（城市安防运营系统）作为第一个应用场景，原因：

- 角色定义明确（调度员、末端用户、系统管理员），角色间冲突丰富
- PRD、场景设计、技术架构文档齐全，可以立即验证 PRD 拆解和协商流程
- 人机协同的四层决策模型（AI 自主→AI 推荐→AI 辅助→纯人类）天然适合用角色协商来定义边界
- 未来可以向客户展示："这个系统不仅是给你用的，设计过程中你的每个岗位都参与了共创"

### 2.3 商业模式演进

**Phase 1（内部工具）：** CrowdDev 作为 CityMatrix 项目的开发辅助工具，验证核心价值。零成本。

**Phase 2（客户协作工具）：** 在向客户交付 CityMatrix 时，CrowdDev 作为"定制化协作平台"的一部分提供。客户的安保运营负责人可以通过 CrowdDev 参与系统的持续优化——他们的反馈被角色 Agent 吸收，驱动系统迭代。嵌入项目服务费。

**Phase 3（独立产品）：** CrowdDev 脱离 CityMatrix，成为通用的"AI 多角色协商设计平台"。面向需要多角色协同设计的团队按月/年订阅收费。

## 三、核心能力框架

### 3.1 四大引擎

```
┌─────────────────────────────────────────────────────────┐
│                    CrowdDev 众开协商引擎                   │
├───────────────┬──────────────┬──────────────┬───────────┤
│   角色引擎     │   协商引擎    │   文档引擎    │  演化引擎  │
│  Role Engine  │  Negotiation │  Doc Engine  │ Evolution │
│               │   Engine     │              │  Engine   │
├───────────────┼──────────────┼──────────────┼───────────┤
│ · 角色档案管理 │ · 多阶段协商  │ · PRD 智能拆解│ · 版本追踪 │
│ · 版本化演化   │ · 冲突识别    │ · 章节预筛    │ · 决策沉淀 │
│ · 跨组织复用   │ · 辩论引导    │ · 共识→PRD回写│ · 角色成长 │
│ · Agent 人格   │ · 共识生成    │ · 多格式支持  │ · 趋势分析 │
└───────────────┴──────────────┴──────────────┴───────────┘
```

### 3.2 角色引擎（Role Engine）

角色不是简单的用户画像，而是一个完整的 Agent 定义：

- **身份层**：姓名、头衔、组织、角色类型
- **职能层**：职责范围、决策权限、操作边界
- **认知层**：专业背景、领域经验、知识结构
- **人格层**：性格特征、沟通风格、风险偏好
- **关切层**：核心诉求、底线约束、利益优先级
- **记忆层**：参与过的讨论、表达过的立场、做出的妥协

角色档案是**可版本化的**。v1.0 的调度员可能只关注"任务完成"，经过多轮讨论后 v1.3 的调度员学会了关注"多任务并发时的优先级冲突"——这个成长过程就是知识沉淀。

**跨组织复用**：同一类角色（如"调度员"）在不同组织中的核心职责相似，差异在于组织特定的参数（权限范围、审批流程、合规要求）。CrowdDev 支持从基础模板派生组织特定角色，共享演化成果。

### 3.3 协商引擎（Negotiation Engine）

协商不是自由讨论，而是有结构的决策过程：

**四步协商流程：**
1. **各方表态**：每个角色独立发表立场（避免从众效应）
2. **冲突分析**：AI 主持人识别矛盾点、分歧根源和焦点问题
3. **辩论回应**：角色针对分歧展开回应（可多轮），人类主持人可引导方向
4. **共识生成**：综合方案 + 各方妥协点 + 遗留分歧 + 可执行的设计决策

**三阶段模型：**

| 阶段 | 讨论焦点 | 输入材料 | 输出物 |
|------|---------|---------|--------|
| 设计期 | "应该怎么做" | PRD文本、需求描述 | 设计决策、验收标准、PRD更新 |
| 验收期 | "做出来的对不对" | 功能描述、操作流程、已知问题 | 改进清单、偏差分析、角色调整 |
| 运营期 | "怎么让它更好" | 运行数据、用户反馈、性能指标 | 策略参数优化、角色演化、新设计议题 |

同一个功能点可以完整经历三个阶段，形成闭环螺旋。

### 3.4 文档引擎（Doc Engine）

文档引擎解决"PRD 太大无法直接讨论"的问题：

- **智能预筛**：AI 扫描全文档，为每个章节评估"多角色讨论价值"（1-10分），标注预期冲突方向
- **议题提取**：从高价值章节中提取具体的协商议题，每个议题带优先级、涉及角色和冲突预判
- **共识回写**：协商达成共识后，自动检查是否存在未解决冲突，生成具体的 PRD 修改条目
- **多格式支持**：支持 Markdown、Word 文档、YAML 策略文件的读取和回写

### 3.5 演化引擎（Evolution Engine）

演化引擎跟踪整个系统的"设计基因"变化：

- **角色版本历史**：每个角色的档案变更记录，形成成长轨迹
- **决策沉淀库**：所有协商的共识、妥协点、遗留分歧，可检索和引用
- **冲突热力图**：哪些功能领域的冲突最多、哪些角色之间的分歧最大
- **趋势分析**：随时间推移，系统设计是趋向更自动化还是更人工干预？角色的决策权限是扩大还是收缩？

## 四、产品演进路线图

### Phase 1：MVP — 核心协商闭环（0-2个月）

目标：验证"AI 角色协商能产出有价值的设计决策"。

核心功能：
- 角色引擎：创建、编辑、版本化管理角色档案
- 协商引擎：完整四步协商流程（表态→分析→辩论→共识）
- 文档引擎：PRD 预筛 + 议题提取 + 共识回写
- 三阶段支持：设计期 / 验收期 / 运营期，各有引导策略模板

验证标准：用 CityMatrix 的 MVP PRD 跑完至少 5 个议题的完整协商，产出的设计决策中至少 60% 被团队采纳。

### Phase 2：深度集成 — 与开发流程打通（2-4个月）

目标：CrowdDev 从独立工具变成开发流程的一部分。

新增功能：
- **Git 集成**：协商共识自动生成 commit message 或 issue
- **YAML 策略文件回写**：共识直接转化为 CityMatrix 的策略参数变更
- **验收模式增强**：接入功能截图和操作流程描述，角色基于实际界面评审
- **批量协商**：支持按 PRD 章节批量创建和运行协商会话
- **协商模板库**：常见议题类型的引导模板（人机分工、操作流程、异常处理等）

### Phase 3：运营闭环 — 数据驱动的持续演化（4-8个月）

目标：角色 Agent 从"设计参与者"进化为"运营分析者"。

新增功能：
- **数据接入**：对接 CityMatrix 的运营数据（任务完成率、AI推荐接受率等）
- **自动议题触发**：当运营数据异常时，自动生成协商议题（如"AI推荐被拒率上升30%，是否需要调整推荐策略？"）
- **角色自动升级**：基于协商历史和运营数据，AI 建议角色档案的更新
- **策略 A/B 测试**：对协商产出的策略变更进行模拟验证后再上线

### Phase 4：产品化 — 通用众开平台（8-12个月）

目标：脱离 CityMatrix，成为可独立销售的产品。

新增功能：
- **多项目支持**：不同项目各自的角色体系和协商空间
- **组织模板**：预置行业角色模板（安防、医疗、工业、零售等）
- **团队协作**：人类团队成员可以直接参与协商（与 AI 角色混合讨论）
- **API 开放**：允许外部系统触发协商、读取共识结论
- **报告生成**：自动生成"系统设计演化报告"，可用于客户汇报和投资人展示

## 五、竞争壁垒分析

### 5.1 为什么不容易被复制

**数据壁垒**：角色档案经过多轮协商的演化，积累了深厚的领域知识和决策模式。这些不是通用 LLM 能产出的——需要在具体场景中反复打磨。

**流程壁垒**：四步协商流程、三阶段模型、预筛→提取→回写的文档处理管道，这些是产品 Know-how 的结晶。

**网络效应**：角色模板可以跨组织复用。使用 CrowdDev 的组织越多，角色模板越丰富、越精准。

### 5.2 与现有工具的差异

| 维度 | 传统用户测试 | AI 设计助手 | CrowdDev |
|------|-------------|------------|----------|
| 反馈来源 | 真人用户 | 单一AI | 多角色AI群体 |
| 反馈时机 | 上线后 | 设计时 | 全生命周期 |
| 冲突处理 | 人工协调 | 无 | 结构化协商 |
| 知识沉淀 | 散落各处 | 无 | 版本化积累 |
| 角色演化 | 无 | 无 | 持续成长 |

## 六、风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| AI 角色不够"真实"，反馈价值低 | 中 | 高 | 持续用真人专家校准角色档案；先从 CityMatrix 的真实角色验证 |
| 协商过程太长，效率不如直接决策 | 中 | 中 | 优化流程：预筛过滤低价值议题、支持批量处理、提供"快速协商"模式 |
| LLM API 成本过高 | 低 | 中 | 优化 prompt 长度；使用更便宜的模型做预筛，高质量模型只用于辩论和共识 |
| 用户难以理解"众开"概念 | 中 | 中 | 从具体场景切入，先让用户体验价值再解释理念；提供 demo 和模板 |

## 七、成功标准

**Phase 1 结束时：**
- CityMatrix PRD 的至少 10 个设计决策点经过 CrowdDev 协商
- 团队采纳率 ≥ 60%
- 至少发现 3 个之前未预见的设计冲突

**Phase 2 结束时：**
- CrowdDev 协商共识可自动同步到代码仓库
- CityMatrix 验收期的功能评审全部通过 CrowdDev 完成
- 角色档案经历至少 3 次版本升级

**Phase 3 结束时：**
- 运营数据异常可自动触发协商议题
- 至少 1 个 CityMatrix 客户通过 CrowdDev 参与系统优化
- 角色 Agent 的建议被运营团队采纳率 ≥ 40%

**Phase 4 结束时：**
- 至少 2 个非 CityMatrix 项目使用 CrowdDev
- 角色模板库包含 3 个以上行业
- 有独立的付费客户
