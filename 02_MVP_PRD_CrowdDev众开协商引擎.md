# CrowdDev 众开协商引擎 — MVP 产品需求文档（PRD）

**版本：** v1.0  
**日期：** 2026年3月  
**MVP 范围：** Phase 1 核心协商闭环  
**预计开发周期：** 6-8周

---

## 一、产品概述

### 1.1 产品定义

CrowdDev 是一个 AI 多角色协商设计工具。用户定义系统中的角色，CrowdDev 让这些角色以 AI Agent 的身份围绕设计议题展开结构化讨论，通过冲突暴露和协商达成共识，驱动系统的设计迭代。

### 1.2 MVP 核心目标

用 CityMatrix（城市安防运营系统）作为首个应用场景，验证三个核心假设：

1. AI 角色能产出有价值的设计反馈（不是泛泛而谈）
2. 结构化协商能暴露真实的设计冲突
3. 协商共识能转化为可执行的 PRD 更新

### 1.3 MVP 不做什么

- 不做团队协作（MVP 阶段为单人使用）
- 不做 Git 集成
- 不做运营数据接入
- 不做角色自动升级（手动编辑升版）
- 不做多项目管理（单项目）

## 二、用户与使用场景

### 2.1 目标用户

MVP 阶段的唯一用户类型：**产品设计者**——负责复杂系统的产品设计，需要从多角色视角审视设计方案的人。

### 2.2 核心使用场景

**场景 A：PRD 设计评审**

产品设计者完成了一份 PRD，想知道不同角色对设计方案的看法。他将 PRD 导入 CrowdDev，系统自动识别出值得讨论的设计决策点，他选择优先级最高的议题发起协商。三个角色 Agent 分别从自己的立场发表观点，协商引擎发现调度员和系统管理员在"一键操作 vs 审计合规"上存在冲突。经过两轮辩论，达成共识："一键确认，但后台自动记录审计日志，管理员可追溯查看"。设计者将这个共识更新到 PRD 中。

**场景 B：功能验收评审**

某个功能开发完成，设计者想从不同角色视角评估实现质量。他创建一个验收期协商会话，描述功能的实际操作流程和已知限制。角色 Agent 基于自己的工作习惯和需求评估这个实现是否合理，指出"追踪画面的加载延迟影响了紧急情况下的使用体验"。设计者获得了具体的改进清单。

**场景 C：角色模型迭代**

经过多轮协商，设计者发现"调度员"角色的关切清单需要更新——之前没有考虑到"多任务并发时的注意力分配"问题。他编辑角色档案，增加这个关切点，角色版本从 v1.0 升级到 v1.1。后续协商中，这个角色会自动关注相关议题。

## 三、功能需求

### 3.1 角色引擎

#### 3.1.1 角色 CRUD

| ID | 需求 | 优先级 |
|----|------|--------|
| RE-01 | 创建角色：填写姓名、头衔、组织、头像（emoji）、职责范围、决策权限、专业背景、性格特征、核心关切 | P0 |
| RE-02 | 编辑角色：修改任意字段，保存时自动升版（v1.0 → v1.1.0），记录变更历史 | P0 |
| RE-03 | 删除角色：需二次确认，关联的历史协商记录保留 | P1 |
| RE-04 | 角色列表：展示所有角色卡片，显示姓名、头衔、版本号、核心关切摘要 | P0 |

#### 3.1.2 角色档案结构

| 字段 | 类型 | 说明 | 必填 |
|------|------|------|------|
| id | string | 唯一标识 | 自动 |
| name | string | 角色姓名（如"李Sir"） | 是 |
| title | string | 角色头衔（如"调度员 Dispatcher"） | 是 |
| organization | string | 所属组织 | 是 |
| avatar | string | emoji 头像 | 是 |
| version | string | 语义化版本号 | 自动 |
| responsibilities | string[] | 职责范围列表 | 是 |
| decisionPowers | string[] | 决策权限列表 | 是 |
| expertise | string[] | 专业背景列表 | 是 |
| personality | string[] | 性格特征列表 | 是 |
| concerns | string[] | 核心关切列表 | 是 |
| history | VersionRecord[] | 版本变更历史 | 自动 |
| createdAt | datetime | 创建时间 | 自动 |

#### 3.1.3 角色 → LLM Prompt 转换

| ID | 需求 | 优先级 |
|----|------|--------|
| RE-05 | 系统根据角色档案自动生成角色扮演的 system prompt，包含所有字段信息 | P0 |
| RE-06 | system prompt 根据当前协商阶段（设计/验收/运营）动态调整引导语 | P0 |
| RE-07 | 角色发言控制在 200 字以内，要求具体、实际，避免泛泛而谈 | P0 |

#### 3.1.4 预置角色

| ID | 需求 | 优先级 |
|----|------|--------|
| RE-08 | 系统预置 CityMatrix 三个角色：调度员（李Sir）、末端用户（张警官）、系统管理员（王工） | P0 |
| RE-09 | 预置角色可被编辑和删除，与自建角色无功能差异 | P0 |

### 3.2 协商引擎

#### 3.2.1 协商会话管理

| ID | 需求 | 优先级 |
|----|------|--------|
| NE-01 | 创建会话：设定议题、背景说明、选择阶段（设计/验收/运营）、选择参与角色（≥2个） | P0 |
| NE-02 | 会话列表：按时间倒序展示，显示议题、阶段标签、参与角色头像、发言数、状态 | P0 |
| NE-03 | 删除会话：需二次确认 | P1 |
| NE-04 | 会话状态流转：created → in_progress → consensus_reached | P0 |

#### 3.2.2 四步协商流程

**Step 1：各方表态**

| ID | 需求 | 优先级 |
|----|------|--------|
| NE-05 | 点击"启动表态"后，每个参与角色依次独立发表对议题的看法 | P0 |
| NE-06 | 每个角色的发言基于其角色档案和当前阶段的引导策略生成 | P0 |
| NE-07 | 发言过程中显示当前正在思考的角色（loading 状态） | P0 |

**Step 2：冲突分析**

| ID | 需求 | 优先级 |
|----|------|--------|
| NE-08 | 点击"分析冲突"后，AI 主持人分析所有发言中的矛盾点 | P0 |
| NE-09 | 分析输出：每个冲突的核心、涉及角色、各方立场、冲突根源 | P0 |
| NE-10 | 提出 1-2 个需要进一步辩论的焦点问题 | P0 |

**Step 3：辩论回应**

| ID | 需求 | 优先级 |
|----|------|--------|
| NE-11 | 点击"辩论回应"后，每个角色针对他人观点和冲突分析进行回应 | P0 |
| NE-12 | 辩论可进行多轮，每轮每个角色各发言一次 | P0 |
| NE-13 | 人类主持人可在辩论前输入引导问题，影响角色的回应方向 | P0 |
| NE-14 | 角色在辩论中可以：反驳、妥协、提出折衷方案、明确底线 | P0 |

**Step 4：共识生成**

| ID | 需求 | 优先级 |
|----|------|--------|
| NE-15 | 点击"寻求共识"后，AI 主持人综合所有讨论生成共识方案 | P0 |
| NE-16 | 共识方案根据阶段不同，使用对应的共识模板（见 3.2.3） | P0 |
| NE-17 | 共识达成后可继续辩论并重新生成共识 | P1 |
| NE-18 | 共识达成后可触发"冲突检查 & PRD更新"（见 3.3） | P0 |

#### 3.2.3 三阶段引导策略

**设计期引导：**

- 角色引导词：关注"应该怎么做"——设计意图、边界情况、人机分工
- 共识模板输出：设计结论 → 各方妥协点 → 设计约束 → PRD更新建议 → 验收标准

**验收期引导：**

- 角色引导词：关注"做出来的对不对"——操作流程是否顺畅、技术妥协是否可接受、是否有预料外的问题
- 共识模板输出：验收判定（通过/有条件通过/不通过）→ 达标项 → 偏差项 → 改进清单 → 新发现 → 角色调整建议

**运营期引导：**

- 角色引导词：关注"怎么让它更好"——基于数据说话、系统瓶颈、AI决策准确率、流程优化
- 共识模板输出：数据洞察 → 优化方案 → 优先级排序 → 策略参数调整 → 角色演化建议 → 新设计议题

#### 3.2.4 消息展示

| ID | 需求 | 优先级 |
|----|------|--------|
| NE-19 | 每条消息显示：发言角色（头像+姓名+头衔）、消息类型标签（立场/回应/分析/共识/PRD更新）、阶段标签、正文内容 | P0 |
| NE-20 | 消息区域可滚动，新消息自动滚动到底部 | P0 |
| NE-21 | 协商引擎（非角色）的消息用不同颜色标识 | P0 |

### 3.3 文档引擎

#### 3.3.1 项目文档读取

| ID | 需求 | 优先级 |
|----|------|--------|
| DE-01 | 系统可读取项目目录中的文档文件（.md、.yaml、.txt） | P0 |
| DE-02 | 文档按章节（标题层级）自动拆分为可独立选择的单元 | P0 |
| DE-03 | 支持手动粘贴文本内容作为替代输入方式 | P1 |

#### 3.3.2 AI 预筛

| ID | 需求 | 优先级 |
|----|------|--------|
| DE-04 | 选择文档后，AI 自动扫描所有章节，为每个章节评估"多角色讨论价值"（1-10分） | P0 |
| DE-05 | 评估输出包含：分数、价值等级（高≥7 / 中5-6 / 低≤4）、一句话评估理由、预判冲突方向 | P0 |
| DE-06 | 结果按分数降序排列，高价值章节自动预选 | P0 |
| DE-07 | 低价值章节（≤4分）显示为半透明，但仍可手动选择 | P1 |
| DE-08 | 提供快捷操作：选高价值 / 选中+高 / 全选 / 清空 | P0 |

#### 3.3.3 议题提取

| ID | 需求 | 优先级 |
|----|------|--------|
| DE-09 | 基于选中的高价值章节，AI 提取具体的协商议题（5-12个） | P0 |
| DE-10 | 每个议题包含：标题、背景说明、涉及角色、预期冲突方向、优先级、所属PRD章节 | P0 |
| DE-11 | 议题按冲突强度排序（最可能产生激烈讨论的排前面） | P0 |
| DE-12 | 用户可勾选/取消议题，批量创建协商会话 | P0 |
| DE-13 | 创建会话时自动关联涉及的角色，阶段默认为"设计期" | P0 |

#### 3.3.4 共识回写

| ID | 需求 | 优先级 |
|----|------|--------|
| DE-14 | 协商达成共识后，可触发"冲突检查 & PRD更新" | P0 |
| DE-15 | AI 首先判断是否仍存在未解决的实质性冲突 | P0 |
| DE-16 | 若冲突已解决，生成具体的 PRD 修改条目（新增/修改/删除，含具体文本和理由） | P0 |
| DE-17 | 若仍有冲突，指出未解决点并建议继续讨论的方向 | P0 |

### 3.4 演化追踪

| ID | 需求 | 优先级 |
|----|------|--------|

## 四、非功能需求

### 4.1 性能

| ID | 需求 |
|----|------|
| NF-01 | 单次 AI 调用响应时间 ≤ 15秒 |
| NF-02 | 一轮完整表态（3个角色）≤ 45秒 |
| NF-03 | PRD 预筛（50个章节）≤ 20秒 |
| NF-04 | 界面交互响应 ≤ 100ms |

### 4.2 可靠性

| ID | 需求 |
|----|------|
| NF-05 | AI 返回的 JSON 解析失败时，自动重试一次，仍失败则显示错误并允许手动重试 |
| NF-06 | 网络中断时不丢失已有数据 |
| NF-07 | 大文档（>50个章节）不导致界面卡顿 |

### 4.3 可用性

| ID | 需求 |
|----|------|
| NF-08 | 首次使用无需教程，通过界面引导即可完成一次完整协商 |
| NF-09 | PRD 拆解流程有清晰的步骤指示（①②③进度条） |
| NF-10 | 协商流程有阶段指示器，用户清楚当前处于哪一步 |

## 五、交互设计要点

### 5.1 整体布局

```
┌─────────────────────────────────────────────────────────────┐
│  ⚡ CrowdDev v1.0     [协商会话] [PRD拆解] [角色引擎] [演化]  │
├───────────────┬─────────────────────────────────────────────┤
│               │                                             │
│  会话侧边栏    │         主内容区                              │
│  （列表+筛选）  │   （协商对话 / PRD拆解 / 角色管理 / 仪表盘）   │
│               │                                             │
│               ├─────────────────────────────────────────────┤
│               │  操作栏（阶段切换、引导输入、操作按钮）          │
└───────────────┴─────────────────────────────────────────────┘
```

### 5.2 关键交互流程

**PRD 拆解流程（三步）：**

```
① 选文档  ────→  ② AI预筛  ────→  ③ 确认议题
   │                  │                  │
 选择项目文档     AI评分排序          勾选并批量
 或手动粘贴      高价值自动预选       创建协商会话
```

**协商流程（四步）：**

```
▶ 启动表态  ──→  🔍 分析冲突  ──→  💬 辩论回应  ──→  🤝 寻求共识
                                    ↑        │
                                    └──多轮───┘
                                                     │
                                               📋 PRD更新
```

### 5.3 视觉规范

- 深色主题为主（与代码工具统一体验）
- 角色用 emoji 头像 + 名称标识
- 三阶段用颜色区分：设计期蓝色、验收期金色、运营期绿色
- 消息类型用左边框颜色区分：立场蓝色、回应金色、分析紫色、共识绿色
- AI 预筛结果用分数条和颜色强度表示讨论价值

## 七、MVP 验收标准

| 编号 | 验收项 | 标准 |
|------|--------|------|
| AC-01 | 角色 CRUD | 可创建、编辑、删除角色，编辑后版本号自动递增 |
| AC-02 | 完整协商 | 可完成从表态→分析→辩论→共识的完整四步流程 |
| AC-03 | 三阶段 | 设计期、验收期、运营期各有不同的引导策略和共识模板 |
| AC-04 | PRD 预筛 | 选择文档后 AI 自动评分排序，高价值章节预选 |
| AC-05 | 议题提取 | 从高价值章节中提取 5+ 个具体协商议题 |
| AC-06 | 共识回写 | 共识达成后可生成具体的 PRD 修改条目 |
| AC-07 | 数据持久化 | 角色和会话数据重启不丢失 |
| AC-08 | CityMatrix 验证 | 用真实 PRD 跑完至少 5 个议题的完整协商 |
